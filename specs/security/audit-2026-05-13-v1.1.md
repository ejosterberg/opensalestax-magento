## v1.0 carry-overs — resolution

### A05 — Backend URL revalidation (was Partial → now Resolved)

**Was:** `osstax/general/api_url` had only the frontend `validate-url` class. A determined admin with JS disabled could persist a malformed URL.

**Now:** Custom backend model `EJOsterberg\OpenSalesTax\Model\Config\Backend\ApiUrl` wired into `etc/adminhtml/system.xml` via `<backend_model>`. Its `beforeSave()`:
- Parses the URL via PHP `parse_url`. Rejects values that lack a scheme or host.
- Enforces a scheme allowlist of `http` and `https`. Rejects `ftp://`, `file://`, `gopher://`, etc.
- Translates validator-thrown `InvalidArgumentException` into `Magento\Framework\Exception\LocalizedException` so the admin form surfaces a translated error.

Logic is delegated to `EJOsterberg\OpenSalesTax\Model\Validator\ApiUrlValidator` — pure PHP, no Magento internals, fully unit-tested.

### A10 — Private-IP allowlist (was Partial → now Resolved)

**Was:** The validator permitted any URL; a merchant who pasted `http://169.254.169.254/` (AWS metadata service) would have it accepted.

**Now:** New admin toggle `osstax/general/restrict_to_public_ips` (default **No**). When set to **Yes**, `ApiUrlValidator` resolves the host (DNS or literal IP) and rejects values that fall in:
- RFC 1918 private ranges (10/8, 172.16/12, 192.168/16)
- Loopback (127/8)
- Link-local (169.254/16)
- Other PHP-reserved ranges per `FILTER_FLAG_NO_RES_RANGE`

Implementation uses `filter_var(..., FILTER_VALIDATE_IP, FILTER_FLAG_NO_PRIV_RANGE | FILTER_FLAG_NO_RES_RANGE)`. The hostname resolver is constructor-injected so tests use deterministic mocks rather than relying on DNS.

The sibling-field read in `ApiUrl::beforeSave()` uses `getFieldsetDataValue('restrict_to_public_ips')` so the admin can toggle the flag and edit the URL in the same save, and validation sees the new flag value rather than the persisted one.

### Default remains permissive

`restrict_to_public_ips` defaults to **No** because the project's reference deployment (constitution §2) puts Magento and the OST engine on the same VM. Merchants in that posture use `http://localhost:8080` or `http://10.x.x.x:8080`, which would fail the public-IP check. The toggle is opt-in defense-in-depth, not a default-on hardening.

## Caveats documented for v1.2

**DNS rebinding is not mitigated.** Save-time IP resolution can return a public address while request-time resolution returns a private one — an attacker who controls the DNS for a hostname can pivot to internal resources after the URL is saved. A full mitigation would:

1. Pin the resolved IP at save time.
2. Force the runtime client to dial that IP (e.g. via `curl --resolve host:port:ip`).

This is queued for v1.2. The v1.1 control surface (save-time validation) is the standard pattern in similar plugins (the [OWASP cheat sheet for SSRF](https://cheatsheetseries.owasp.org/cheatsheets/Server_Side_Request_Forgery_Prevention_Cheat_Sheet.html) recommends both layers; we have layer 1).
