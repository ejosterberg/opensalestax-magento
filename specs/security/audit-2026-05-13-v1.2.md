## What changed

### Save-time IP pinning

When `restrict_to_public_ips` is on, the backend model now does two things in sequence:

1. **`beforeSave`** runs the existing v1.1 validation: parse URL, enforce scheme allowlist, resolve host, verify the IP is public (not RFC 1918 / loopback / link-local / reserved).
2. **`afterSave`** persists the resolved IP to `osstax/general/api_url_pinned_ip` in the same scope (`default`, `websites`, or `stores`) via `Magento\Framework\App\Config\Storage\WriterInterface`. The pin is cleared (delete) when the URL is empty or the toggle is off.

### Request-time IP forcing

`OstaxClient::configureCurl` now calls a new `applyPinnedIp` helper that, when a pin is present:

1. Parses the configured URL to extract host + port (default 80 for http, 443 for https).
2. Sets `CURLOPT_RESOLVE` to the `host:port:pinned-ip` triple.

cURL then bypasses DNS at request time and dials the pinned IP directly. TLS verification still applies — for HTTPS engines, the certificate is verified against the original hostname, so an attacker who tricks the merchant's DNS into pointing at their server cannot impersonate the engine.

## Constraints / non-goals

- **DNS rebinding mitigation is opt-in.** Toggle defaults to off so the constitution §2 reference deployment (Magento + OST on the same VM via `http://localhost:8080` or `http://10.x.x.x:8080`) keeps working without configuration. Merchants whose engine runs on a separate host turn it on.
- **Pinning is keyed by `(host, port)`.** Changing the URL's port without changing the host requires a re-save to refresh the pin. (Beneficial side effect: a stale pin against a no-longer-valid port simply fails the cURL connect → fail-soft to Magento's tax tables, never to an unintended target.)
- **Engine IP rotation requires re-save.** Merchants on cloud platforms where engine IPs change (Heroku, EC2 reboots) must either leave the toggle off or re-save the URL after each rotation. Documented in the field's admin comment and CHANGELOG.
- **Pinned IP is not surfaced in the admin UI.** It is visible in `core_config_data` for ops investigation. A read-only display field is a v1.3 polish candidate.
- **TLS cert pinning is not added.** Standard cURL cert verification (`CURLOPT_SSL_VERIFYPEER` default = on) is sufficient given the hostname check still happens against the original URL host. Cert pinning would be a useful v1.3+ enhancement for merchants on mutual TLS engines.
